Stratora v2.1.7 fixes a regression in the auto-generated site dashboard: the Network Topology panel on every site's dashboard now renders its map correctly on fresh deploys, on new sites, and on sites where the panel was previously stuck showing "No map selected." No upgrade steps required — existing dashboards repair themselves on the next page load.

<!-- truncate -->

## What was broken

Since v2.1.3 introduced auto-generated site dashboards alongside auto-generated topology maps, a timing issue could leave a dashboard's "Network Topology" panel with no map linked. The panel would render "No map selected" forever, even though the site's topology map existed and was fully populated.

Three root causes combined to cause this:

- On a fresh install, the default-site setup path bypassed the topology auto-generation chain. The dashboard was created, but the topology map generation was gated on "site has at least one node" — and the default site had zero nodes at install time.
- When the first node was later added, the dashboard-regen and topology-map-regen jobs fired as sibling background tasks with no ordering. The dashboard job would usually win, see no map yet, and persist an empty map reference on the topology panel. The map was created a moment later — but no further dashboard regen was triggered, so the panel kept its empty reference indefinitely.
- Dashboards that got caught in this race had no way to self-correct. Customers had to manually re-trigger a regen (by adding/removing a node) to get the map linked.

## The four fixes

**1. Fresh installs follow the same auto-generation path as any other site creation.** The default site gets its dashboard immediately and its topology map as soon as the first node arrives.

**2. Topology regeneration runs before dashboard regeneration.** The race is gone. Every site and node operation now wraps both jobs so topology runs first and the dashboard always sees the fresh map.

**3. Dashboard read path self-heals empty map references.** Any auto-generated site dashboard that was persisted with no map linked will render correctly on the next page load: the dashboard response resolves the live map at serve time. The next normal regeneration event makes the fix permanent. Customers don't need to do anything — the panel just starts working again.

**4. Auto-generated topology map IDs are now deterministic.** The map ID is derived from the site ID. If the auto-map row is ever deleted and regenerated, the new row receives the *same* ID — so any dashboard that was carrying the old map ID reference still resolves correctly instead of becoming a dangling pointer.

## Operator note

The deterministic-ID behavior in fix #4 is intentional. Hard-deleting an auto-generated topology map row (for example, to force a clean layout regeneration) used to produce a new random ID; in v2.1.7 it produces the same ID as before. Dashboards referencing the old ID continue to work, which is the win, but operators who were relying on delete-to-get-fresh-ID should instead edit the map in the Map Editor.

## Bundled components

No Agent or Collector changes shipped in this release. The Server MSI embeds the unchanged Agent 2.1.6 (Windows), Agent 1.2.1 (Linux), and Collector 2.1.6 components, preserving the validated in-place upgrade path introduced in v2.1.6.
