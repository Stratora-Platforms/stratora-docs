Stratora v2.1.5 is an important fix release that resolves two issues preventing SNMP monitoring from working correctly on fresh deployments. All discovered network devices should now begin collecting metrics immediately after the setup wizard completes.

<!-- truncate -->

### Device Template Path Fix

On fresh installations, the MSI installer was placing device template files (YAML configurations for Palo Alto firewalls, Aruba APs, Cisco switches, QNAP NAS units, and others) in the wrong directory. The backend expected templates at `config\templates\devices\` but the installer was writing them to `config\devices\`, causing all template lookups to fail silently. As a result, 10 out of 12 common device types produced no SNMP polling configuration, and discovered devices showed as "Offline" with "Never" for last metrics — even though the collector service was running normally.

This release corrects the installer packaging so templates are placed at the expected path. All supported device types now generate proper SNMP input blocks in the collector's Telegraf configuration.

### SNMP Credential Resolution Fix

Devices without a matching device template (such as generic SNMP devices or ESXi hosts without a template assignment) use a fallback SNMP configuration path. Previously, this fallback defaulted the SNMP community string to `"public"` regardless of which credential was assigned to the device. The fallback now reads from the same encrypted credential store used by template-based devices, ensuring the correct community string is always used.

### Upgrade Notes

Existing installations can upgrade by running the new `Stratora-Server-2.1.5.msi` installer over the existing installation. This is a **strongly recommended upgrade** for any deployment installed with v2.1.2 through v2.1.4, as those versions were affected by the template path issue.

After upgrading, restart the backend service to regenerate collector configurations. Devices should begin reporting metrics within 2-3 minutes.
