Stratora v2.1.9 fixes a long-standing bug where the system tray icon failed to auto-start at logon on every release from 2.0 through 2.1.8. The fix migrates tray registration from a per-user `HKCU\Run` custom action to a declarative per-machine `HKLM\Run` registry value, so the tray now launches for every interactive user on every release target. Orphan registry values left behind by prior installers are cleaned up on upgrade.

## The auto-start bug

The Agent and Collector MSIs registered the tray icon for auto-start by invoking `stratora-tray.exe -install-startup` from a WiX custom action. That action ran under the MSI's deferred execution context with `Impersonate="no"` — which in Windows terms means it executes as `LocalSystem`. The tray binary then called the Win32 registry API with `HKEY_CURRENT_USER`, which for a SYSTEM-token process resolves to `HKEY_USERS\S-1-5-18` — the SYSTEM profile's hive, not any real user's.

The net result: every prior 2.x install wrote a `StratoraTray` Run value into a hive that no interactive user ever reads from. The tray never fired at logon. The one case it appeared to "work" was when an admin installed the MSI interactively through the wizard — a second custom action (`LaunchTray`, correctly impersonated) launched the tray once for that admin's session, and a self-heal path inside the tray added a stray `HKCU\Run` entry under that specific admin's hive as a side effect. That entry only ever fired for that one admin, on that one box.

## The fix

v2.1.9 replaces the custom-action registration with a declarative `<RegistryValue>` element on the `TrayExe` component in both the Agent and Collector WiX projects:

- Agent MSI writes `HKLM\Software\Microsoft\Windows\CurrentVersion\Run\StratoraTray` pointing at `C:\Program Files\Stratora\stratora-tray.exe`
- Collector MSI writes `HKLM\Software\Microsoft\Windows\CurrentVersion\Run\StratoraTrayCollector` pointing at `C:\Program Files\Stratora\Agent\stratora-tray.exe`

`HKLM\Run` fires for every interactive user at logon, so any admin (local or domain) with RDP or console access to the box gets a tray icon showing the agent's connection state. The distinct value names (`StratoraTray` vs `StratoraTrayCollector`) mean a host with both MSIs historically installed doesn't have the Names collide.

## Removing the self-healing write path

The tray binary itself used to carry a self-registration path: on every non-flag launch, `ensureStartupRegistered()` would open `HKCU\Run` and write a `StratoraTray` value if one was missing. The intent was a per-user fallback. In practice, with the WiX CA moving to HKLM, this fallback would have re-introduced the dual-writer condition the audit flagged — every user who ever opened the tray would stamp an HKCU entry that shadowed the HKLM one, producing two tray icons per logon.

2.1.9 deletes the entire self-registration path: the `registerStartup` / `unregisterStartup` / `ensureStartupRegistered` functions are gone, and the `-install-startup` / `-uninstall-startup` command-line flags on `stratora-tray.exe` are removed. The installer is now the sole writer of the Run key. If you have scripted deployments that previously called `stratora-tray.exe -install-startup` out of band, those invocations no longer do anything and should be removed.

## Single-instance guard for the upgrade window

Upgraded hosts from 2.1.8 may carry a leftover `HKCU\Run\StratoraTray` value in the installing admin's hive (from the now-removed self-heal path). That value still fires at logon, so on the first post-upgrade logon for that admin, both the leftover HKCU entry and the new HKLM entry would launch the tray — two icons in the system tray.

To catch this, 2.1.9 adds a named-mutex single-instance guard to `main()`. The second launch sees the mutex already held, closes its handle, and exits silently. The user sees one tray icon, as intended. The guard uses `Local\` scope (session-local), so each interactive user on a multi-admin server still gets their own tray — the mutex only dedupes duplicates inside the same session.

## Orphan cleanup on upgrade

The pre-2.1.9 SYSTEM-hive write — `HKEY_USERS\S-1-5-18\...\Run\StratoraTray` — is a registry turd. The value is harmless (no user ever reads from that hive at logon) but it's still a record of the old bug. 2.1.9's installer adds two deferred custom actions sequenced after `RemoveExistingProducts` that run `reg.exe delete` on both `StratoraTray` and `StratoraTrayCollector` under `HKU\S-1-5-18\...\Run`. Cleanup runs on upgrade installs only; fresh installs have nothing to clean up and full uninstalls already remove the new HKLM value declaratively.

One caveat: the per-user HKCU leftover on the installing admin's session is *not* cleaned up automatically. The 2.1.9 installer only touches the SYSTEM hive. The single-instance mutex ensures the leftover doesn't cause user-visible double-launches, so it's cosmetic. Operators who want a spotless registry can remove it manually:

```powershell
reg delete "HKCU\Software\Microsoft\Windows\CurrentVersion\Run" /v StratoraTray /f
```

## Upgrade path

Upgrading from 2.1.8 or earlier is a standard MSI in-place upgrade. The Server MSI bundles fresh 2.1.9 Agent and Collector MSIs; no per-component download or staging is required. After the upgrade completes, log off and back on to trigger `HKLM\Run` — the tray will appear in every interactive user's system tray on first logon.
