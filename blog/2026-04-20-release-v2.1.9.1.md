Stratora v2.1.9.1 is a focused patch release fixing three pre-GA blockers surfaced during 2.1.9 agent-deployment testing. Agent install commands now correctly use the server FQDN regardless of how the admin is browsing the UI, collector failover and discovery fallback now work on every deployment rather than just one specific hostname, and the node detail API no longer reports "Not assigned" for collectors that were actually assigned. No new features; no schema changes; an in-place MSI upgrade is all that's needed.

## Why a 2.1.9.1 rather than rolling forward to 2.1.10

v2.1.9 shipped to the releases page but was never publicly announced. Three blockers showed up on the very first customer-representative agent deployment test, each latent until the specific conditions that exposed them lined up. Two of them are silent-until-observed — administrators can run for weeks without hitting the failure mode. Rather than sit on 2.1.9 while 2.1.10 accumulates, we cut a narrow bug-only patch. 2.1.9.1 is effectively the first GA of the 2.1.9 line.

## Fix 1 — Agent install commands now use the configured FQDN

The setup wizard's "Deploy Agents" step always got this right: it reads the configured FQDN from `/api/v1/system/info` (which surfaces `server.external_url` from the backend config) and templates it into the generated `msiexec ... SERVERURL="https://<fqdn>"` command. The standalone Agents page and the Deploy Collector modal did not — they used `window.location.origin`, i.e. whatever URL the admin had in their browser at the moment they copied the command.

That works by accident when the admin is browsing via the FQDN. It doesn't work when the admin is browsing via the raw Windows hostname (e.g. `https://server-hostname/agents` on a LAN where DNS isn't set up yet, or via an IP for a first-time deployment). The generated command then points the fresh agent at a URL that doesn't match NGINX's `server_name` or the TLS certificate. The agent partially enrolls via NGINX's default-server fallback, the cert check skates through, and subsequent heartbeats land in a zombie "Disconnected" state that looks like a network bug but is really a name-resolution mismatch on the agent's initial self-report.

The fix extracts the FQDN-derivation into a shared `useConfiguredServerUrl` React hook and wires it into all three surfaces. The existing "server address not yet configured" warning banner that the wizard showed when `external_url` wasn't set has been propagated to the Agents page and the Deploy Collector modal, so administrators running the setup wizard out of order get the same clear signal everywhere.

If you have agents currently stuck in the disconnected state from a 2.1.9 deployment that used the wrong SERVERURL, re-running the MSI install with the correct URL (from the post-2.1.9.1 Agents page) will re-enroll cleanly.

## Fix 2 — Collector failover and discovery fallback now work on every deployment

The 2.1.8 collector-failover work introduced an `is_local` flag on `remote_components` to identify "the collector running on the Stratora server itself" — the fallback target when a remote preferred collector goes offline for 10+ minutes. Migration 129 seeded this flag on existing deployments by UPDATE-ing the one row whose hostname matched a specific dev-environment value. That literal was correct on exactly one host.

On every other deployment, the UPDATE matched zero rows and the flag was never set. `GetLocalCollector()`, the DB query that drives failover and discovery's last-resort collector assignment, filters on `is_local = true` — so on every deployment except that one dev box, it returned nil. Both code paths logged a warning and gracefully continued, which meant the failure was completely silent to administrators. The latent consequence: a preferred collector going offline for the 10-minute failover window would *not* fail over to the local collector; sites would just stop receiving data. Discovery's fallback path for devices imported without a site or collector hint also stopped working.

The fix replaces the migration's hostname literal with a startup flagger (`services.EnsureLocalCollectorFlagged`) that runs after migrations and before any service that calls `GetLocalCollector`. It reads `os.Hostname()` and flags the matching `remote_components` row. Idempotent; re-applied on every boot; does nothing when the flag is already correct.

Administrators who saw `[CollectorFailover] Cannot fail over site 'X': no local collector found` messages in the backend log will see them stop after the first 2.1.9.1 startup. Administrators running on DJN-DC-DEV01 (the one deployment migration 129's literal was written against) will see no change — the flag was already correct there.

## Fix 3 — Node detail API now reports assigned collectors

The node list endpoint (`GET /api/v1/nodes`) has always joined the `collector_targets` junction table via a `ct_first` lateral subquery to surface each node's `remote_collector_id` and `remote_collector_name`. The single-node endpoint (`GET /api/v1/nodes/:id`) did not carry the same JOIN. Every call returned `remote_collector_id: null`, regardless of the actual DB state.

The user-visible symptom: opening a node's detail page, clicking Edit, selecting a collector from the dropdown, and saving. The save hits a different endpoint (`PUT /collectors/:id/assign`) that correctly persists to `collector_targets`. React Query invalidates the single-node cache and refetches — and the refetch sees the broken `null`. The dropdown snaps back to "Not assigned." The collector is actually assigned in the database the whole time; the UI just can't see it.

The fix ports the same `ct_first` subquery into `GetNode()`'s SELECT. The update also re-reads via `GetNode()` at the return of `UpdateNode()`, so PUT responses reflect the post-update state rather than the pre-update local struct. No frontend changes.

## Upgrade path

In-place MSI upgrade from v2.1.9 or earlier. No schema migration other than a no-op marker. The startup flagger runs once on first 2.1.9.1 boot and the repair takes effect immediately.

If you're currently on a pre-2.1.9 release, upgrade through v2.1.9.1 directly — there's no reason to stop at 2.1.9.

The v2.1.9 release remains on the releases page for audit transparency but is not recommended for new deployments. Every 2.1.9.1 fix supersedes it.
